# Raj Intelligence Desk - Conversation Contract And State Plan

## Purpose

This document starts the new design phase where Raj AI Concierge is being converted from a prompt-heavy assistant into a controlled conversation engine.

The goal is to define and lock the approach piece by piece before executing code changes.

## Why This Redesign Is Needed

Testing showed that raw chat history and prompt instructions are not enough to control a guided professional conversation.

The assistant repeatedly showed these issues:

- Generic overview loops.
- Repeated follow-up options.
- Weak continuity across turns.
- Poor handling of subject-depth prompts.
- Model-generated options that did not advance the conversation.
- Route confusion between work experience, AI depth, hiring, collaboration, and interview evaluation.

The conclusion is clear:

- Code must control the conversation path.
- The model should only write the answer body.
- Lambda must validate and assemble the final response.

## Step 1 - Conversation Contract

### Request Contract

Frontend should send this structure to Lambda:

```json
{
  "session_id": "RID-20260613-ABCD",
  "question": "Which signals predict failure?",
  "chat_history": [
    {"role": "user", "text": "Explore BPCL reliability work."},
    {"role": "assistant", "text": "Previous assistant answer."}
  ],
  "conversation_state": {
    "current_route": "WORK_EXPERIENCE",
    "current_topic": "BPCL",
    "topic_turn_count": 1,
    "total_turn_count": 3,
    "covered_experiences": ["BPCL"],
    "last_options": [
      "Explore BPCL reliability work.",
      "Which signals predict failure?"
    ],
    "selected_option": null
  }
}
```

### Response Contract

Lambda should return this structure:

```json
{
  "status": "success",
  "assistant": "Raj AI Concierge",
  "profile": "Rajesh Arigala",
  "answer": "- Bullet one.\n- Bullet two.\n- Bullet three.",
  "options": [
    "Explore Medtronic adoption systems.",
    "Which signals predict adoption friction?"
  ],
  "conversation_state": {
    "current_route": "AI_TECHNICAL_DEPTH",
    "current_topic": "BPCL",
    "topic_turn_count": 2,
    "total_turn_count": 4,
    "covered_experiences": ["BPCL"],
    "last_options": [
      "Explore Medtronic adoption systems.",
      "Which signals predict adoption friction?"
    ],
    "selected_option": null
  },
  "model_id": "amazon.nova-pro-v1:0",
  "sources": []
}
```

## Locked Design Decisions For Step 1

- `answer` contains only the 3 main answer bullets.
- Follow-up options are not embedded inside `answer`.
- `options` is a separate array generated by code.
- `conversation_state` is sent and returned every request.
- `chat_history` remains separate from `conversation_state`.
- `conversation_context` can be supported temporarily for backward compatibility.

## Why Both Chat History And State Are Needed

`chat_history` is raw memory. It helps with tone, wording continuity, and recent reference resolution.

`conversation_state` is structured memory. It controls route, topic, turn counts, covered experiences, selected options, and next choices.

The system should not rely on chat history alone.

## Step 2 Preview - State Rules

The next step is to finalize state rules before implementation.

State rules must define:

- What counts as a route change.
- What counts as a topic change.
- How selected options are detected.
- How free-typed questions override or continue state.
- How topic turn count increments.
- How covered experiences are tracked.
- When the system moves from one experience to the next.
- When comparison options appear.
- How subject-depth options are selected.

No code should be executed for the redesign until these rules are agreed.
